# Helix integration for Keleusma

> **Navigation**: [Editors](../README.md) | [Repository Root](../../README.md)

Helix uses tree-sitter for syntax highlighting. A full Helix integration consists of a tree-sitter grammar in a separate repository (conventionally `tree-sitter-keleusma`), Helix highlight queries in `runtime/queries/keleusma/highlights.scm`, and a `languages.toml` entry that points Helix at both.

The V0.2.0 release ships only the `languages.toml` snippet. The tree-sitter grammar and the highlight queries are not yet implemented. This README documents the gap and the planned path so an early adopter can either wait for the grammar or contribute one.

## Files

| File | Status |
|------|--------|
| [`languages.toml`](./languages.toml) | File-type registration. Carries the `.kel` extension, line and block comment tokens, four-space indentation. Sufficient for Helix to recognise the file as Keleusma; not sufficient for syntax highlighting until a grammar is in place. |

## Installation

Append the contents of [`languages.toml`](./languages.toml) to your Helix configuration:

```sh
cat editors/helix/languages.toml >> ~/.config/helix/languages.toml
```

Open a `.kel` file. Helix should report the language as `keleusma` in the status line and apply the comment-token and indentation settings, but text will appear unhighlighted until a tree-sitter grammar is available.

## Planned: tree-sitter grammar

The conventional location is a separate repository `tree-sitter-keleusma` published under the same GitHub organisation as the parent crate. The grammar consists of:

- `grammar.js` defining the parser
- `src/parser.c` generated by `tree-sitter generate`
- `queries/highlights.scm` mapping parse nodes to Helix highlight scopes
- `queries/indents.scm`, `queries/textobjects.scm`, `queries/locals.scm` for the corresponding Helix features

Once the grammar exists, this directory's `languages.toml` snippet uncomments the `[[grammar]]` block pointing at the grammar repository.

## Why a tree-sitter grammar is not bundled with V0.2.0

Writing a tree-sitter grammar is roughly the same effort as writing the parser itself: every production needs to be defined twice (once in `src/parser.rs`, once in `grammar.js`), and the two must remain in lockstep across language revisions. The maintenance cost is real and is best taken on once the surface language stabilises.

Helix's roadmap and the Zed editor both consume tree-sitter grammars; both editors are early-adopter targets for Keleusma. Contribution would benefit both editors plus any Neovim users running `nvim-treesitter`.

## Alternative: Vim runtime under Helix

Helix does not consume Vim syntax files. The `editors/vim/` directory of this repository covers Vim and Neovim users; it is not usable from Helix.
