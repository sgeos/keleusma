# Measured Cost Models

> **Navigation**: [keleusma-bench](../README.md)

Pre-generated cost model fragments produced by `keleusma-bench` on specific host architectures. Each fragment is a Rust source file that hosts include into their build via `include!` to obtain a measured `CostModel` for use in WCET analysis on that architecture.

## Contents

| Fragment | Counter | Host architecture |
|----------|---------|-------------------|
| [aarch64_apple_darwin.rs](./aarch64_apple_darwin.rs) | AArch64 CNTVCT_EL0 | Apple Silicon Macs running macOS |

Additional fragments are generated by running `keleusma-bench` on the target host. See the [parent README](../README.md) for the generation procedure.

## How to use a fragment in a host crate

```rust
use keleusma::{CostModel, VALUE_SLOT_SIZE_BYTES};

include!(concat!(env!("CARGO_MANIFEST_DIR"), "/path/to/aarch64_apple_darwin.rs"));

let model: CostModel = MEASURED_COST_MODEL;
```

Hosts that want to use the measured model instead of the bundled `NOMINAL_COST_MODEL` pass the measured one through whichever Keleusma APIs accept `&CostModel`.

## Soundness

The bench tool's methodology approximates pipelined cycles through best-of-many measurements with warm caches. Categories whose representative opcodes cannot be exercised in isolation by the current bench harness (`Yield` requires a Stream chunk; `Call` requires a multi-chunk module) fall back to a *scaled* nominal value (the runtime's bundled `nominal_op_cycles` multiplied by the maximum measured-to-nominal ratio observed across measured categories). This keeps the units consistent across the model: every reported value is in CPU cycles. Without the scaling the fallback would mix nominal relative-weight units with measured CPU-cycle units, producing an incoherent model.

Measured values in the fragment header are CPU cycles, computed by scaling the raw counter delta through the counter's `cpu_cycles_per_count` ratio. The ratio is the assumed CPU clock divided by the counter tick rate. On AArch64 platforms where CNTVCT_EL0 runs at a fraction of CPU clock speed (24 MHz on Apple Silicon, far below CPU clock), one counter tick is approximately 134 CPU cycles on a 3.228 GHz core. The fragment header records the counter tick frequency, the assumed CPU clock, and the resulting scale factor for transparency.

The `KELEUSMA_BENCH_CPU_HZ` environment variable overrides the assumed CPU clock when regenerating a fragment. The default is documented in `keleusma-bench/src/counter.rs` (`DEFAULT_ASSUMED_CPU_HZ`); operators on a host whose CPU clock differs should set the variable to obtain accurate CPU-cycle values for their host.

The generated fragments are not certified hard-real-time bounds. They are best-effort estimates suitable for soft-real-time analysis and order-of-magnitude WCET. The CPU-cycle scaling depends on an assumed clock frequency that may not match the host's instantaneous frequency under thermal throttling, frequency scaling, or core type heterogeneity (P-core vs. E-core on Apple Silicon). For certified bounds, use the bundled `NOMINAL_COST_MODEL` with a verified-cost VM, or use a third-party static-analysis toolchain (aiT, Bound-T) on the native lowering.

## Regeneration

To regenerate the fragment for a given host, on that host:

```sh
cargo run --release --bin keleusma-bench -- \
    --output keleusma-bench/measured_cost_models/<triple>.rs
```

where `<triple>` matches the Rust target triple (e.g., `aarch64-apple-darwin`, `x86_64-unknown-linux-gnu`).

The fragment header records the cycle counter used. If the counter changes between runs (e.g., a Linux build on Apple Silicon uses a different counter than macOS), the comment block on the file documents which counter produced the values.
